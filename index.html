but anyway, i wanna lay out some of my approaches for web design, for the interested and for myself.

1. go for the easiest solution. the great thing about web dev versus large coding projects is that when you make haunted festering code, it is contained to a tiny section that cannot impact other parts of your site.
if i want an image to float beside text, i'm going to jam it in there in whatever way i can that fulfils my criteria. right now, my criteria explicitly doesn't care about mobile users or reactive page design. i actually kinda hate reactive design.
an example of that, my last webpage just put its style in the header rather than an external file, and the styling was horrible. if you put that on any other page it would explode. but it works for that page!
this does NOT go for things i will refuse. common javascript code is the main example.

2. as little backend as possible. if backend is there, it should provide a service that physically cannot be provided through static file serving.
my reason: preservation. static file serving is simple and easy, backend is custom. if your backend goes down, unless you fix it, that is gone forever.
i find it also forces me to come up with more creative solutions on backend problems

3. related, as little reliance as possible. again, preservation. i like to store my fonts on my server, i store my audio and databases on my server, i don't link to external stylesheets or libraries.
i feel this also keeps it simple!
